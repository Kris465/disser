# План интеграции в инфраструктуру

**Проект:** Система изолированных сессий рабочих станций  
**Организация:** ООО «Академия ТОП»  
**Статус:** В разработке  
**Версия:** 0.2

---

## 1. Цели интеграции

1. Развёртывание сервера управления в филиале (г. Королёв)
2. Подключение до 100 рабочих станций в 5 компьютерных классах
3. Обеспечение бесперебойной работы сессий для дизайнеров и разработчиков
4. Минимизация простоя во время внедрения

## 2. Сетевая архитектура

### 2.1. Изолированный сегмент сети

**Важно:** Рабочие станции студентов должны находиться в отдельном, изолированном сегменте сети.

```
┌──────────────────────────────────────────────────────────┐
│  Интернет                                                │
└──────────────────────────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────┐
│  Основной периметр (Firewall/Router)                     │
│  - Блокировка исходящего трафика со станций студентов    │
│  - Доступ только к студенческому порталу (HTTPS)         │
└──────────────────────────────────────────────────────────┘
                           │
        ┌──────────────────┴──────────────────┐
        │                                     │
        ▼                                     ▼
┌──────────────────┐              ┌──────────────────────────┐
│  Административная│              │  Изолированный сегмент   │
│  сеть            │              │  (студенческие станции)  │
│  - Серверы       │              │  - VLAN для студентов    │
│  - Рабочие места │              │  - Нет доступа к другим  │
│    преподавателей│              │    сегментам сети        │
│  - Администраторы│              │  - Доступ только к:      │
└──────────────────┘              │    • Серверу сессий      │
                                  │    • Студенческому       │
                                  │      порталу (браузер)   │
                                  └──────────────────────────┘
```

### 2.2. Требования к сетевой изоляции

| Компонент | Требование | Обоснование |
|-----------|------------|-------------|
| VLAN | Отдельная VLAN для студенческих станций | Логическая изоляция трафика |
| Firewall | Блокировка межсегментного доступа | Защита от горизонтального перемещения |
| DNS | Локальный DNS для сегмента | Нет доступа к внешним ресурсам |
| Интернет | Только через прокси (белый список) | Контроль исходящих соединений |
| Доступ к серверу | Только порты 5000, 8000, 6080 | Минимальные необходимые порты |

### 2.3. Схема адресации

| Сегмент | Подсеть | Назначение |
|---------|---------|------------|
| Сервер сессий | `192.168.1.100/24` | Registry + API |
| Студенческие станции | `192.168.1.0/24` | Компьютерные классы |
| Преподавательские | `192.168.2.0/24` | Отдельная сеть |
| Административная | `192.168.10.0/24` | Серверы и ИТ-персонал |

## 3. Этапы интеграции

### Этап 1: Подготовка инфраструктуры

**Срок:** 1-2 дня

#### 1.1. Выделение серверного оборудования

| Требование | Минимум | Рекомендуется | Статус |
|------------|---------|---------------|--------|
| CPU | 4 ядра | 8 ядер | ☐ |
| RAM | 8 ГБ | 16 ГБ | ☐ |
| Disk | 50 ГБ SSD | 200 ГБ SSD | ☐ |
| Network | 1 Gbps | 1 Gbps | ☐ |
| ОС | Windows 10/11 или Linux | Linux (Ubuntu 22.04) | ☐ |

#### 1.2. Сетевая настройка

- [ ] Назначить статический IP серверу (например, `192.168.1.100`)
- [ ] Открыть порты в firewall:
  - `5000` — Docker Registry
  - `8000` — FastAPI
- [ ] Проверить доступность из всех компьютерных классов

#### 1.3. Подготовка хранилища

- [ ] Выделить раздел/диск для образов (~50-200 ГБ)
- [ ] Настроить мониторинг свободного места

### Этап 2: Развёртывание сервера

**Срок:** 1 день

#### 2.1. Установка Docker

**Linux:**
```bash
curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh
sudo usermod -aG docker $USER
```

**Windows:**
- Установить Docker Desktop
- Включить WSL 2 backend

#### 2.2. Развёртывание компонентов

```bash
cd server

# Сборка и запуск
docker-compose up -d --build

# Проверка
docker-compose ps
curl http://localhost:8000/health
```

#### 2.3. Сборка и загрузка образов

```bash
# Базовый образ
cd images/base-session
docker build -t base-session:latest .

# Образ дизайнера
cd ../designer-base
docker build -t designer-base:latest .

# Образ разработчика
cd ../developer-base
docker build -t developer-base:latest .

# Загрузка в registry (используйте IP сервера для продакшена)
docker tag designer-base:latest 192.168.1.100:5000/designer-base:latest
docker tag developer-base:latest 192.168.1.100:5000/developer-base:latest
docker push 192.168.1.100:5000/designer-base:latest
docker push 192.168.1.100:5000/developer-base:latest
```

> **Примечание:** Для локального тестирования на одном компьютере используйте `localhost:5000` вместо IP сервера.

#### 2.4. Проверка сервера

```bash
# Проверка API
curl http://192.168.1.100:8000/health
curl http://192.168.1.100:8000/image/designer
curl http://192.168.1.100:8000/image/developer

# Проверка registry
curl http://192.168.1.100:5000/v2/_catalog
```

### Этап 3: Пилотное развёртывание (1 класс)

**Срок:** 2-3 дня

#### 3.1. Выбор пилотного класса

- [ ] Класс с 20-25 рабочими станциями
- [ ] Наличие Windows 10/11 (64-bit)
- [ ] Сетевая доступность до сервера

#### 3.2. Настройка рабочих станций

**На каждой станции:**

```powershell
# 1. Установка Docker Desktop
.\station\scripts\install-docker.ps1

# 2. Перезагрузка
Restart-Computer

# 3. Установка Python (если нет)
# Скачать с python.org и установить с опцией "Add to PATH"

# 4. Настройка автозапуска
.\station\scripts\setup-autostart.ps1
```

#### 3.3. Настройка лаунчера

Открыть `station\launcher\config.py` и указать:

```python
SESSION_SERVER = "http://192.168.1.100:8000"
```

#### 3.4. Тестирование

- [ ] Лаунчер запускается автоматически
- [ ] Выбор сессии работает
- [ ] noVNC открывается в браузере
- [ ] Приложения запускаются внутри сессии
- [ ] Контейнер удаляется после завершения

#### 4.2. Настройка автозапуска (без домена)

**Вариант A: Локальная задача в планировщике**

На каждой станции выполняется скрипт:
```powershell
.\station\scripts\setup-autostart.ps1
```

Скрипт создаёт:
- Задачу в планировщике на запуск при входе любого пользователя
- Ярлык в папке автозагрузки
- Задачу на cleanup при выключении

**Вариант B: Локальная групповая политика (gpedit.msc)**

1. Открыть `gpedit.msc`
2. Перейти: `Конфигурация пользователя → Конфигурация Windows → Сценарии (вход/выход)`
3. Добавить скрипт входа:
   ```powershell
   python D:\repos\disser\station\launcher\main.py
   ```

**Вариант C: Реестр (автозапуск для всех пользователей)**

1. Открыть `regedit`
2. Перейти: `HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\CurrentVersion\Run`
3. Создать строковый параметр:
   - Имя: `SessionLauncher`
   - Значение: `python D:\repos\disser\station\launcher\main.py`

**Блокировка Docker CLI:**
```powershell
# Запрет запуска docker.exe для пользователей
icacls "C:\Program Files\Docker\Docker\resources\docker.exe" /deny "Пользователи:(RX)"
```

#### 4.3. Настройка сетевой изоляции

**На коммутаторе:**
```bash
# Пример для Cisco
vlan 100
 name STUDENT-STATIONS
interface range GigabitEthernet1/0/1-24
 switchport mode access
 switchport access vlan 100
```

**На firewall:**
| Правило | Источник | Назначение | Порт | Действие |
|---------|----------|------------|------|----------|
| Доступ к серверу сессий | VLAN 100 | 192.168.1.100 | 5000, 8000, 6080 | Разрешить |
| Доступ к порталу | VLAN 100 | Студенческий портал | 443 | Разрешить |
| Остальной исходящий | VLAN 100 | Любой | Любой | Запретить |
| Межсегментный | VLAN 100 | Другие VLAN | Любой | Запретить |

## 4. Риски и план их минимизации

| Риск | Вероятность | Влияние | Меры |
|------|-------------|---------|------|

```bash

docker-compose logs -f api

# Список образов
docker images | grep base-session

# Остановка всех сессий

docker system prune -a
```
