# Архитектура системы изолированных сессий

**Статус:** Архитектурное описание
**Дата:** 25 марта 2026

---

## 1. Назначение системы

Система обеспечивает изолированные эфемерные сессии для рабочих станций в образовательной организации (ООО «Академия ТОП»).

**Ключевые требования:**
- Пользователь запускает рабочую станцию → выбирает тип сессии (дизайнер/разработчик)
- Контейнер разворачивается локально на станции (не на сервере)
- Сервер только оркестрирует: хранит образы, отдаёт метаданные
- Сессия уничтожается при выключении ПК
- Пользователь не имеет доступа к хостовой ОС и не знает о контейнере
- Работа на старом парке машин (Windows 10/11)
- Простое развёртывание любым сисадмином

## 2. Архитектура

### 2.1. Компоненты

```
┌──────────────────────────────────────────────────────────┐
│  Сервер управления (филиал, локальная сеть)              │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Docker Registry                                   │  │
│  │  - Хранение образов (designer-base, developer-base)│  │
│  │  - Версионирование образов                         │  │
│  └────────────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────────────┐  │
│  │  HTTP API (FastAPI)                                │  │
│  │  - GET /image/{type} → metadata (name, hash, url)  │  │
│  │  - GET /health → статус сервера                    │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
                           │
                           │ HTTP (локальная сеть)
                           ▼
┌──────────────────────────────────────────────────────────┐
│  Рабочая станция №N (Windows 10/11)                      │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Лаунчер (Python + Tkinter)                        │  │
│  │  - Выбор сессии: [Дизайнер] [Разработчик]          │  │
│  │  - Запрос к серверу → metadata образа              │  │
│  │  - Запуск контейнера                               │  │
│  └────────────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────────────┐  │
│  │  Docker Контейнер (изолированная сессия)           │  │
│  │  - noVNC + Xvfb (GUI через браузер)                │  │
│  │  - Приложения (по типу образа)                     │  │
│  │  - Полноэкранный режим (киоск)                     │  │
│  │  - Изоляция (security options)                     │  │
│  │  - Уничтожение при shutdown (--rm)                 │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

### 2.2. Поток сессии

```
1. Включение ПК → загрузка Windows
2. Автозапуск лаунчера (Tkinter, полный экран)
3. Пользователь выбирает тип сессии
4. Лаунчер → запрос к серверу: GET /image/{type}
5. Сервер → возвращает metadata образа
6. Лаунчер проверяет локальный кэш Docker
7. Запуск контейнера (если нет кэша → скачивание)
8. noVNC открывается в браузере (полный экран)
9. Пользователь работает в изолированной среде
10. Выключение ПК → контейнер удаляется (--rm)
```

### 2.3. Обработка ошибок

| Сценарий | Действие |
|----------|----------|
| Сервер недоступен | Запуск локального кэшированного образа |
| Нет кэша + сервер недоступен | Заглушка: «Неисправность сети» |
| Контейнер не запускается | Лог ошибки + код ошибки |
| Пользователь закрыл браузер | noVNC перезапускается |
| Принудительный shutdown | Docker удаляет контейнер (--rm) |

## 3. Технические решения

### 3.1. Сервер управления

| Компонент | Технология |
|-----------|------------|
| ОС | Windows 10/11 или Linux |
| Docker Registry | Официальный образ `registry:2` |
| HTTP API | FastAPI (Python) |

**Требования к серверу:**
| Компонент | Минимум | Рекомендуется |
|-----------|---------|---------------|
| CPU | 4 ядра | 8 ядер |
| RAM | 8 ГБ | 16 ГБ |
| Disk | 50 ГБ SSD | 200 ГБ SSD |
| Network | 1 Gbps | 1 Gbps |

### 3.2. Рабочая станция

| Компонент | Требование |
|-----------|------------|
| ОС | Windows 10/11 (64-bit) |
| Docker | Docker Desktop |
| Python | 3.8+ |
| RAM | 8+ ГБ (рекомендуется 16 ГБ) |
| CPU | 4+ ядра |

### 3.3. Образы

| Образ | Приложения | Размер |
|-------|------------|--------|
| `designer-base` | Krita, GIMP, Inkscape, Darktable, Kdenlive, Blender, Firefox | ~3.5 ГБ |
| `developer-base` | VS Code, PHP, Python, Node.js, C++, Chrome, Apache, Nginx | ~3 ГБ |

**Базовый слой (общий):**
- Ubuntu 22.04 LTS
- Xvfb + XFCE
- noVNC + websockify

### 3.3.1. Обоснование Linux для дизайнеров

**Аппаратные требования:**
| Критерий | Linux (контейнер) | Windows (хост) |
|----------|-------------------|----------------|
| RAM | 1-2 ГБ | 4-8 ГБ |
| CPU | 2 ядра | 4+ ядра |
| Disk | ~3.5 ГБ | ~10-20 ГБ |

**Экономия (500 ПК):**
| Статья | Linux | Windows |
|--------|-------|---------|
| Adobe Creative Cloud | $0 | ~$120,000/год |
| Лицензии Windows Pro | $0 | ~$70,000 |
| Замена парка ПК | $0 | ~$250,000 |
| **Итого** | **~$370,000+** | — |

**Аналоги Adobe:**
| Adobe | Linux-аналог | Покрытие |
|-------|--------------|----------|
| Photoshop | Krita + GIMP | 90% |
| Illustrator | Inkscape | 95% |
| Lightroom | Darktable | 85% |
| Premiere Pro | Kdenlive | 80% |
| After Effects | Blender | 75% |

### 3.4. Безопасность и изоляция

| Мера | Реализация |
|------|------------|
| Изоляция от хоста | `--cap-drop=ALL`, `--read-only`, `--tmpfs` |
| Автоудаление сессии | `docker run --rm` |
| Полноэкранный режим | Браузер в режиме киоска |
| Блокировка остановки | Лаунчер скрывает Docker CLI |

## 4. Нагрузка и производительность

### 4.1. Профиль нагрузки

| Параметр | Значение |
|----------|----------|
| Количество станций | до 100 (5 классов × 20-25 ПК) |
| Пиковые запуски | каждые 1.5 часа |
| Одновременных запусков в пик | 2-100 |
| Длительность пика | 5-10 минут |

### 4.2. Трафик

| Сценарий | Трафик на станцию | Общий (100 станций) |
|----------|-------------------|---------------------|
| Первый запуск | ~2.5 ГБ | ~250 ГБ |
| Повторный запуск (кэш) | ~0 ГБ | ~0 ГБ |
| Обновление образа | ~100-500 МБ | ~10-50 ГБ |

### 4.3. Производительность сервера

| Операция | Время | RPS |
|----------|-------|-----|
| GET /image/{type} | <10 мс | 1000+ |
| GET /health | <5 мс | 1000+ |

## Приложения

### A. Пример запроса к серверу

**Запрос:**
```http
GET /image/designer HTTP/1.1
Host: 192.168.1.100:8000
```

**Ответ:**
```json
{
  "image_name": "designer-base",
  "version": "v1.2",
  "hash": "sha256:abc123...",
  "registry_url": "http://192.168.1.100:5000",
  "size_bytes": 2621440000
}
```

### B. Пример запуска контейнера

```bash
docker run --rm \
  --cap-drop=ALL \
  --read-only \
  --tmpfs /tmp \
  --tmpfs /run \
  --name session_%COMPUTERNAME%_%DATE% \
  -p 5900:5900 \
  designer-base:v1.2
```
