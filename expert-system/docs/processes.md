# Процессы управления проектом

**Версия:** 2.0
**Дата:** 16 мая 2026

---

## 1. Процедура управления изменениями (Change Request)

### 1.1. Назначение

Процедура управления изменениями обеспечивает контролируемое внесение изменений в содержание, требования, архитектуру или правила базы знаний в ходе реализации проекта. Любое изменение, не предусмотренное утверждённым планом, проходит через данную процедуру.

### 1.2. Инициирование изменения

Запрос на изменение (Change Request) может быть инициирован любым участником проекта при обнаружении:

- нового параметра безопасности, не учтённого в базе знаний
- ошибки в существующем правиле (некорректное условие, вес, уровень риска)
- изменения требований спонсора
- изменения конфигурации основной системы (проект ООО «Академия ТОП»)
- результатов тестирования, выявивших ложные срабатывания или пропуски

### 1.3. Форма запроса на изменение

| Поле | Описание |
|------|----------|
| **ID запроса** | CR-YYYY-NNN (год + порядковый номер) |
| **Дата** | Дата подачи запроса |
| **Инициатор** | ФИО, роль |
| **Описание изменения** | Что именно предлагается изменить |
| **Обоснование** | Почему изменение необходимо |
| **Влияние на содержание** | Какие компоненты/правила/документы затрагиваются |
| **Влияние на сроки** | Оценка дополнительного времени |
| **Влияние на качество** | Оценка влияния на метрики точности |
| **Приоритет** | Высокий / Средний / Низкий |

### 1.4. Процесс рассмотрения

```
1. [Инициатор]  → Заполняет форму Change Request
2. [Руководитель проекта] → Оценивает влияние на сроки и содержание
3. [Эксперт ИБ] → Оценяет влияние на корректность правил (если изменение в БЗ)
4. [Руководитель проекта] → Принимает решение: Утвердить / Отклонить / Отложить
5. [Исполнитель] → Реализует изменение (если утверждено)
6. [QA-инженер] → Верифицирует изменение (повторный прогон тест-кейсов)
7. [Руководитель проекта] → Закрывает запрос, обновляет документацию
```

### 1.5. Журнал изменений

Все запросы фиксируются в журнале изменений (Change Log), который является обязательным артефактом при закрытии проекта. Формат журнала:

| ID | Дата | Инициатор | Описание | Решение | Дата реализации |
|----|------|-----------|----------|---------|-----------------|
| CR-2026-001 | (дата) | (ФИО) | (описание) | Утверждено / Отклонено | (дата) |

## 2. Процесс формализации экспертных знаний

### 2.1. Назначение

Процесс формализации экспертных знаний определяет порядок сбора, структурирования и верификации знаний экспертов по информационной безопасности для последующего включения в базу знаний экспертной системы.

### 2.2. Источники знаний

| Источник | Тип знаний | Метод сбора |
|----------|-----------|-------------|
| Специалист по ИБ | Правила оценки защищённости, веса угроз | Интервью, семинары |
| Администратор ВДИ | Типичные конфигурации, известные уязвимости | Интервью, анализ логов |
| Документация основного проекта | Архитектура, security options, потоки данных | Анализ документации |
| Стандарты и методики (СТР-К, ФСТЭК, NIST) | Требования к защите информации | Анализ нормативных документов |

### 2.3. Этапы формализации

#### Этап 1. Подготовка (до начала сбора)

```
1. Составить перечень параметров безопасности (на основе онтологии домена)
2. Подготовить шаблон правила (IF-THEN формат)
3. Подготовить справочник угроз (модель угроз)
4. Назначить дату и время семинара с экспертом
```

#### Этап 2. Сбор знаний (семинар / интервью)

```
1. Для каждого параметра безопасности:
   a. Эксперт формулирует условие уязвимости:
      "При каком значении параметра возникает риск?"
   b. Эксперт определяет уровень риска:
      CRITICAL / HIGH / MEDIUM / LOW
   c. Эксперт формулирует рекомендацию:
      "Что нужно сделать для устранения?"
2. Для каждой угрозы из модели угроз:
   a. Проверить, что угроза покрыта хотя бы одним правилом
   b. При необходимости — добавить новое правило
3. Эксперт оценивает относительную важность правил (для определения весов)
```

#### Этап 3. Структурирование

```
1. Зафиксировать каждое правило в стандартном формате:
   - ID, название, условия, уровень риска, вес, угроза, сообщение, рекомендация
2. Проверить полноту: все ли угрозы из модели покрыты правилами
3. Проверить непротиворечивость: нет ли конфликтующих правил
4. Присвоить веса на основе экспертных оценок
```

#### Этап 4. Верификация

```
1. Эксперт проверяет каждое правило на корректность
2. Эксперт проверяет полноту покрытия модели угроз
3. Эксперт проверяет рекомендации на применимость
4. Эксперт подписывает протокол верификации базы знаний
```

### 2.4. Шаблон протокола верификации эксперта

```
ПРОТОКОЛ ВЕРИФИКАЦИИ БАЗЫ ЗНАНИЙ

Дата: ___________
Эксперт: ___________ (ФИО, должность, квалификация)

1. Проверено правил: ___ из ___
2. Угроз покрыто: ___ из ___
3. Замечания: _________________________________
4. Рекомендации: ______________________________

Заключение:
  [ ] База знаний верифицирована, готова к использованию
  [ ] Требуется доработка (см. замечания)

Подпись эксперта: ___________
```

### 2.5. Итеративная доработка

База знаний является живым артефактом. После тестирования на эталонных стендах возможна итеративная доработка:

1. Анализ результатов тестирования (TP/TN/FP/FN)
2. Выявление правил с ложными срабатываниями (FP) или пропусками (FN)
3. Корректировка условий, весов или порогов
4. Повторная верификация экспертом
5. Повторное тестирование

Доработка выполняется через процедуру Change Request (раздел 1 данного документа).
