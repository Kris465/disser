## 1. Ресурсы для реализации прототипа (Этап 3 — Исполнение)

### 1.1. Требуемые специалисты

| Роль | Кол-во | Занятость | Длительность | Требования к квалификации |
|------|--------|-----------|--------------|--------------------------|
| **Python-разработчик (основной)** | 1 | 100% | 3 недели | Python 3.10+, опыт работы с Docker API, знание основ информационной безопасности |
| **Python-разработчик (UI)** | 1 | 50% | 1 неделя | Python, Flask/FastAPI, HTML/CSS, Jinja2 |
| **DevOps-инженер** | 1 | 25% | 1 неделя | Docker, Docker Compose, настройка тестовых стендов |

**Итого трудозатраты на реализацию:** ~5.5 человеко-недель.

### 1.2. Технологический стек

| Компонент | Технология | Обоснование выбора |
|-----------|-----------|-------------------|
| Язык реализации | Python 3.10+ | Единый стек с основным проектом (сервер управления уже на Python/FastAPI) |
| Работа с Docker | docker-py (SDK) | Программный сбор конфигураций контейнеров без shell-команд |
| Формат базы знаний | YAML (PyYAML) | Человекочитаемый формат, легко редактировать без перекомпиляции |
| Веб-интерфейс | FastAPI + Jinja2 | Единый стек с сервером управления, минимум зависимостей |
| Генерация отчётов | Jinja2 → HTML → WeasyPrint | Шаблонный подход, экспорт в PDF |
| Конфигурация | Pydantic | Валидация входных данных, типобезопасность |
| Логирование | logging (stdlib) | Стандартный модуль, без внешних зависимостей |
| Тестирование | unittest (stdlib) | Стандартный модуль, достаточен для 15 правил |

### 1.3. Аппаратные ресурсы

| Ресурс | Назначение | Требования |
|--------|-----------|------------|
| **Рабочая станция разработчика** | Разработка и отладка | Windows 10/11, 16 ГБ RAM, Docker Desktop, Python 3.10+ |
| **Тестовый сервер** | Развёртывание тестового стенда | 8 ГБ RAM, Docker Engine, 20 ГБ диска |
| **Тестовые рабочие станции (2-3 шт.)** | Эталонные стенды (Perfect/Default/Broken) | Аналогичны целевым станциям из основного проекта |

### 1.4. Программные ресурсы (бесплатные/открытые)

| Компонент | Лицензия | Назначение |
|-----------|----------|------------|
| Python 3.10+ | PSF License | Язык реализации |
| Docker Community Edition | Apache 2.0 | Контейнеризация тестовых стендов |
| FastAPI | MIT | Веб-фреймворк для Dashboard |
| PyYAML | MIT | Парсинг базы знаний |
| WeasyPrint | BSD | Генерация PDF-отчётов |

**Стоимость лицензий:** ₽0 (все компоненты с открытым исходным кодом).

### 1.5. Структура каталогов реализованного проекта

При реализации компания должна создать следующую структуру:

```
expert-system/
├── config/
│   └── rules.yaml              # База знаний (правила, веса, множители)
├── src/
│   ├── __init__.py
│   ├── main.py                 # Точка входа
│   ├── knowledge_base/
│   │   ├── __init__.py
│   │   ├── rules.py            # Загрузка и валидация правил из YAML
│   │   └── ontology.py         # Онтология домена
│   ├── inference_engine/
│   │   ├── __init__.py
│   │   └── engine.py           # Forward chaining + расчёт индекса
│   ├── data_collection/
│   │   ├── __init__.py
│   │   ├── docker_parser.py    # Парсер конфигураций Docker
│   │   ├── network_parser.py   # Парсер сетевых настроек
│   │   └── auth_parser.py      # Парсер аутентификации
│   └── ui/
│       ├── __init__.py
│       ├── app.py              # FastAPI-приложение
│       └── templates/          # HTML-шаблоны (Jinja2)
│           ├── dashboard.html
│           └── report.html
├── tests/
│   ├── __init__.py
│   ├── test_rules.py           # Тесты базы знаний
│   ├── test_inference.py       # Тесты механизма вывода
│   └── test_parsers.py         # Тесты парсеров
└── requirements.txt            # Зависимости Python
```

## 2. Ресурсы для тестирования и валидации (Этап 4)

### 2.1. Требуемые специалисты

| Роль | Кол-во | Занятость | Длительность | Ответственность |
|------|--------|-----------|--------------|-----------------|
| **Специалист по ИБ (эксперт)** | 1 | 50% | 1 неделя | Валидация правил, подтверждение корректности выводов, подписание протокола валидации |
| **QA-инженер** | 1 | 100% | 1 неделя | Подготовка стендов, прогон тест-кейсов, расчёт метрик |
| **Администратор ВДИ** | 1 | 25% | 1 неделя | Подготовка эталонных конфигураций, развёртывание тестовых стендов |
| **Python-разработчик** | 1 | 25% | 1 неделя | Исправление дефектов, выявленных при тестировании |

**Итого трудозатраты на тестирование:** ~2.5 человеко-недели.

### 2.2. Тестовая инфраструктура

| Ресурс | Назначение | Требования |
|--------|-----------|------------|
| **Тестовый стенд «Perfect»** | Идеальная конфигурация | Docker с полным набором security options, настроенный firewall, MFA |
| **Тестовый стенд «Default»** | Конфигурация по умолчанию | Docker с минимальными настройками безопасности |
| **Тестовый стенд «Broken»** | Намеренно уязвимая | Docker без изоляции, открытый clipboard, без MFA |
| **Сетевая инфраструктура** | Эмуляция сегмента сети | VLAN, firewall rules, шифрование трафика |

### 2.3. Критерии входа в тестирование

| # | Критерий | Проверяемый артефакт |
|---|----------|---------------------|
| 1 | Все модули реализованы и прошли модульное тестирование | Исходный код + отчёт о модульных тестах |
| 2 | База знаний загружена и валидирована | Файл config/rules.yaml |
| 3 | Тестовые стенды развёрнуты и задокументированы | Описание конфигураций стендов |
| 4 | План тестирования утверждён | docs/test-plan.md |

### 2.4. Критерии выхода из тестирования

| # | Критерий | Целевое значение |
|---|----------|------------------|
| 1 | Precision | ≥ 95% |
| 2 | Recall | ≥ 95% |
| 3 | F1-мера | ≥ 95% |
| 4 | Все критические дефекты устранены | 0 открытых critical |
| 5 | Эксперт ИБ подписал протокол валидации | Подпись |

|------|--------|-----------|--------------|-----------------|
| **Руководитель проекта** | 1 | 50% | 1 неделя | Координация, приёмка артефактов, организация демонстрации |
| **Спонсор (CISO / Lead Architect)** | 1 | 25% | 0.5 недели | Рассмотрение документации, подписание акта приёмки |
| **Технический писатель** | 1 | 50% | 1 неделя | Подготовка руководства администратора, оформление итогового отчёта |

**Итого трудозатраты на закрытие:** ~2 человеко-недели.

### 3.2. Перечень документов для приёмки

| # | Документ | Назначение | Ответственный |
|---|----------|-----------|---------------|
| 1 | **Акт приёмки** | Формальное подтверждение соответствия результата требованиям устава | Руководитель проекта + Спонсор |
| 2 | **Руководство администратора** | Инструкция по установке, настройке и эксплуатации системы | Технический писатель |
| 3 | **Итоговый отчёт о тестировании** | Результаты тестирования на эталонных стендах, метрики качества | QA-инженер |
| 4 | **Протокол валидации эксперта ИБ** | Подтверждение корректности правил и выводов | Специалист по ИБ |
| 5 | **Итоговый отчёт по проекту** | Описание достигнутых результатов, отклонений от плана, извлечённых уроков | Руководитель проекта |
| 6 | **Реестр извлечённых уроков (Lessons Learned)** | Фиксация опыта для будущих проектов | Руководитель проекта |
| 7 | **Акт передачи исходного кода** | Передача кода и конфигураций заказчику | Разработчик + Руководитель |
| 8 | **Журнал изменений (Change Log)** | Все запросы на изменения, принятые в ходе проекта | Руководитель проекта |

### 3.3. Содержание ключевых документов

#### Руководство администратора (структура)

```
1. Введение
   1.1 Назначение системы
   1.2 Требования к среде
2. Установка
   2.1 Установка Python и зависимостей
   2.2 Настройка конфигурации
   2.3 Запуск системы
3. Настройка базы знаний
   3.1 Формат правил (YAML)
   3.2 Добавление/изменение правил
   3.3 Настройка весов и порогов
4. Эксплуатация
   4.1 Запуск оценки
   4.2 Интерпретация результатов
   4.3 Экспорт отчётов
5. Устранение неисправностей
   5.1 Типичные ошибки
   5.2 Логи и диагностика
6. Приложения
   6.1 Справочник правил
   6.2 Справочник угроз
```

#### Итоговый отчёт по проекту (структура)

```
1. Резюме проекта
   1.1 Цели и достигнутые результаты
   1.2 Ключевые метрики (KPI)
2. Выполнение плана
   2.1 Отклонения от расписания
   2.2 Отклонения от содержания
   2.3 Управление рисками (что сработало, что нет)
3. Результаты тестирования
   3.1 Метрики качества (Precision, Recall, F1)
   3.2 Сравнение с целевыми значениями
4. Извлечённые уроки
   4.1 Что удалось хорошо
   4.2 Что можно улучшить
5. Рекомендации
   5.1 Дальнейшее развитие системы
   5.2 Масштабирование
```

## 5. Зависимости и ограничения

### 5.1. Внешние зависимости

| Зависимость | Описание | Влияние при невыполнении |
|-------------|----------|--------------------------|
| Доступ к эксперту ИБ | Необходим для формализации правил и валидации | Задержка на 1–2 недели, снижение качества правил |
| Тестовый стенд | Требуется оборудование для 3 стендов | Невозможность валидации на эталонах |
| Совместимость с основным проектом | ЭС должна работать с Docker Engine из проекта ООО «Академия ТОП» | Необходима адаптация парсеров |

### 5.2. Ограничения

- Бюджет: только трудозатраты (лицензий ₽0)
- Сроки: 4 недели на реализацию + тестирование + приёмку
- Стек: только Python + открытые библиотеки
- Офлайн-работа: система должна функционировать без внешних сервисов
