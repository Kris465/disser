## 2. Бизнес-обоснование

### 2.1. Проблема

Система изолированных сессий рабочих станций (проект ООО «Академия ТОП») обеспечивает эфемерные контейнерные сессии для компьютерных классов. Однако текущий проект не содержит подсистемы автоматической оценки уровня защищённости сегмента сети. Администратор вынужден вручную проверять десятки параметров безопасности:

- настройки изоляции контейнеров (`--cap-drop`, `--read-only`, `--tmpfs`)
- политики аутентификации (MFA, парольная политика)
- сегментацию сети (VLAN, firewall)
- мониторинг и аудит

Ручная проверка занимает значительное время, подвержена человеческим ошибкам и не обеспечивает непрерывного контроля.

### 2.2. Целесообразность

Автоматизация оценки защищённости позволит:

1. **Сократить время аудита** с нескольких часов до минут.
2. **Исключить субъективность** — оценка формализованных правил.
3. **Обеспечить непрерывный мониторинг** — система может работать в фоне.
4. **Снизить риск утечек данных** — своевременное обнаружение уязвимостей.
5. **Соответствовать модели Zero Trust** — постоянная верификация, а не разовый аудит.

## 3. Цели проекта

### 3.1. Стратегическая цель

Создать экспертную систему, которая автоматически оценивает уровень защищённости сегмента сети, использующего модель изолированных сессий рабочих станций, и выдаёт обоснованный вердикт с рекомендациями.

### 3.2. Измеримые цели (SMART)

| Цель | Метрика | Целевое значение |
|------|---------|------------------|
| Автоматический сбор параметров конфигурации | % параметров, собираемых автоматически | ≥ 90% |
| Формализация экспертных знаний | Количество правил в базе знаний | ≥ 15 правил |
| Точность оценки | Precision / Recall на эталонных стендах | ≥ 95% |
| Время формирования отчёта | Секунды | ≤ 30 сек |
| Покрытие модели угроз | % угроз из реестра, покрытых правилами | 100% |

## 4. Ключевые требования

### 4.1. Функциональные требования

1. **Сбор данных:** автоматический сбор параметров конфигурации Docker-контейнеров, сетевых настроек, политик аутентификации.
2. **База знаний:** хранение формализованных правил оценки защищённости в формате IF-THEN.
3. **Механизм вывода:** прямая цепочка рассуждений (forward chaining) для применения правил к собранным данным.
4. **Расчёт индекса:** вычисление числового индекса защищённости (0–100) на основе весов сработавших правил.
5. **Верификация:** классификация вердикта (Безопасно / Требуется доработка / Уязвимо).
6. **Отчётность:** формирование отчёта с детализацией по сработавшим правилам и рекомендациями.

### 4.2. Требования к среде

- Локальная работа на сервере управления или выделенной рабочей станции
- Язык реализации: Python 3.10+
- Отсутствие обязательных внешних зависимостей (офлайн-работа)

## 9. Критерии успеха проекта (по PMBOK)

### 9.1. Критерии содержания (Scope)
- [ ] Все 10 угроз из модели угроз покрыты правилами базы знаний

- [ ] Precision на эталонных стендах ≥ 95%

- [ ] Отчёт формируется за ≤ 30 секунд
